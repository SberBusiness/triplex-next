# Коммиты и ветки — Triplex-Next

---

## Именование веток

**Формат:** `TRIPLEX-XXX-короткое-описание`

- Строчные буквы, слова через дефис.
- Всегда начинается с номера тикета.

```text
TRIPLEX-844-ai-ready-design-system
TRIPLEX-901-add-badge-component
TRIPLEX-912-fix-button-focus-style
```

**Исключения:**
- Релизная ветка: `prepare-release-XX` (например, `prepare-release-2`)
- Перелитие main → release-0: выполняется мейнтейнерами, формат произвольный

---

## Структура веток и релизов

| Ветка | React | Версии пакета | Описание |
|---|---|---|---|
| `main` | React 18 | `1.xx` | Основная ветка разработки |
| `release-0` | React 17 | `0.xx` | Поддержка React 17, синхронизируется с main |

**Правило:** все фича-ветки создаются от `main`. Изменения попадают в `release-0` через перелитие — вручную мейнтейнерами. Агент не должен создавать ветки от `release-0` и не должен мерджить в неё напрямую.

## Формат коммитов

**Формат:** `TRIPLEX-XXX Краткое описание`

```text
TRIPLEX-844 Добавить AI-Ready документацию для компонентов
TRIPLEX-901 Добавить компонент Badge с темами и размерами
TRIPLEX-912 Исправить стиль фокуса для Button в теме Link
```

Если нужно уточнение — добавляй через двоеточие:

```text
TRIPLEX-844 AI-Ready Design System: фундамент и пилот Button
TRIPLEX-901 Badge: добавить тему DANGER и размер SM
```

**Правила:**
- Все коммиты в ветке начинаются с одного номера тикета.
- Первая строка — не длиннее 72 символов.
- Тело коммита опционально (для сложных изменений).
- Язык: русский или английский — не смешивать в одном коммите.

## Когда запускать линтер

**Вручную запускать не нужно.** Линтер запускается автоматически при `git commit`:

1. `husky` перехватывает коммит.
2. `lint-staged` запускает на staged-файлах:
   - `eslint --fix` — автоисправление ESLint-ошибок
   - `prettier --write` — форматирование кода
3. Исправленные файлы добавляются обратно в staging.

**Если `lint-staged` упал:**
- Прочитай ошибку — это ESLint-нарушение, которое нельзя автофиксировать.
- Исправь вручную.
- Пере-stage: `git add <files>`.
- Повтори коммит.

**Ручной запуск (если нужно проверить до коммита):**
```bash
npx eslint --fix src/components/Button/Button.tsx
npx prettier --write src/components/Button/Button.tsx
```

## Что не коммитить

| Файл/папка | Причина |
|---|---|
| `__screenshots__/` | Только из Docker/CI. Если нужно — подтвердить в pre-commit hook |
| `.mcp.json` | Персональный токен Figma — в `.gitignore` |
| `.env.local` | Секреты — в `.gitignore` |
| `node_modules/`, `dist/`, `storybook-static/` | В `.gitignore` |

Если в ходе работы появились новые локальные, временные, генерируемые или
машино-специфичные файлы:
- проверь `git status`, не попали ли они в untracked/staged
- если такие файлы не должны храниться в репозитории, добавь подходящее правило в `.gitignore`
- не коммить подобные файлы "на всякий случай"

## Когда коммитить и пушить

**Агент не создаёт коммиты самостоятельно.** После завершения задачи — остановиться и показать результат разработчику. Коммит делается только по явной просьбе.

**Агент не пушит ветку самостоятельно.** Пуш — только по явной просьбе разработчика.

---

## PR-воркфлоу

**Агент пушит ветку — разработчик создаёт PR.**

```bash
git push -u origin TRIPLEX-XXX-feature-name
```

После пуша разработчик сам открывает PR через GitHub UI или `gh pr create`.

**Исключение:** если разработчик явно просит агента создать PR, используй:

```bash
gh pr create \
  --title "TRIPLEX-XXX Краткое описание" \
  --body "## Что сделано\n- ...\n\n## Как проверить\n- ..."
```

## Release notes

При добавлении или изменении публичного поведения компонента — обнови файл release notes текущей версии:

```
stories/release-notes/v1/<текущая версия>.mdx
```

**Что писать:**

- Новый prop: «Добавлено свойство `X` для ...»
- Изменение поведения: «Изменён/Исправлен ...»
- Удалённый prop (breaking change): выноси в секцию `<Heading>Ломающие изменения</Heading>`

**Формат записи:**

```mdx
**ComponentName**

- Добавлено свойство `propName` для описания.
```

Компоненты внутри секции — в алфавитном порядке.

Не нужно добавлять release notes для:
- изменений только в stories/тестах/документации
- рефакторинга без изменения публичного API

## Чек-лист перед пушем

- [ ] Линтер проходит (автоматически при коммите)
- [ ] Unit-тесты проходят: `npm run test-unit`
- [ ] Нет случайно добавленных файлов (`git status`)
- [ ] Для новых локальных/временных/генерируемых файлов проверено, не нужно ли добавить правило в `.gitignore`
- [ ] Нет скриншотов из macOS в `__screenshots__/`
- [ ] Release notes обновлены (если изменился публичный API компонента)
