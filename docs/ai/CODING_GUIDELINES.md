# Coding Guidelines — Triplex-Next

Краткая сводка обязательных правил. Детальные конвенции — в отдельных файлах:

- **Стек и архитектура:** `docs/ai/CONTEXT.md`
- **Стили кода:** `docs/ai/codestyle.md`
- **Тестирование:** `docs/ai/tests.md`
- **Stories:** `docs/ai/stories-guide.md`
- **Коммиты и ветки:** `docs/ai/commits.md`

Если другой документ формулирует правило иначе, этот файл и более детальный
профильный гайд имеют приоритет над entrypoint-файлами.

---

## Архитектура и структура

- Каждый компонент — в своей папке `src/components/{ComponentName}/`.
- Минимальный ожидаемый набор файлов для публичного компонента: `{ComponentName}.tsx`, `index.ts`, стили и тесты по локальному паттерну компонента.
- `index.ts` — только barrel export, без логики. Всё, что там было — должно оставаться (breaking change удалить экспорт).
- Все компоненты экспортируются через `src/index.ts`.
- Не у каждого компонента уже есть `{ComponentName}-AI.md`. Если файла нет, опирайся на `CONTEXT.md`, исходники, stories и тесты и не изобретай новую структуру без задачи.

## TypeScript

- `strict: true` — никаких исключений.
- Запрещено: `any`, `!` (non-null assertion) без комментария, `as Type` без необходимости.
- Именование: `I` (interface), `E` (enum), `T` (union type), `UPPER_SNAKE_CASE` (константы).
- Интерфейсы props — экспортируются (часть публичного API).
- Для несовместимых визуальных или поведенческих вариантов предпочитай discriminated union, а не набор пересекающихся boolean-props.
- Новый prop делай optional по умолчанию, если требование не диктует обратное.

## React

- `forwardRef` — обязателен на всех UI-компонентах.
- `clsx` — для className. Никогда не конкатенировать строки.
- `className` и `...rest` — прокидываются в корневой элемент.
- `React.memo` — только по результатам профилирования.
- Корневой DOM-элемент, семантика и ref-target — часть контракта компонента. Менять их без явной причины нельзя.
- Если компонент поддерживает управляемый state, контракт должен быть предсказуемым: `value`/`open` + callback, а `defaultValue`/`defaultOpen` — только когда компонент действительно умеет жить в uncontrolled-режиме.

## Стилизация

- Только LESS CSS Modules (`.module.less`). Нет inline styles в компонентах.
- Классы — camelCase. Цвета и размеры — только через CSS-переменные токенов.
- Нет `!important` без комментария.

## Accessibility

- Семантические HTML-элементы (`button` вместо `div` с `onClick`).
- Keyboard navigation: Tab, Enter, Escape, стрелки — где уместно.
- Accessible name и `aria-*` поведение — часть публичного UX-контракта интерактивного компонента.

## Тестирование

- Unit-тесты: Vitest + Testing Library. Покрывай новый функционал.
- Новый компонент: полное покрытие (все props, forwardRef, события).
- Visual regression: автоматически из stories. Baseline только из CI/Docker.
- E2E: Playwright, только для критичных флоу.
- В unit-тестах проверяй поведение и публичные состояния. CSS-классы допустимы только когда они кодируют публичный вариант или состояние компонента.
- Не требуй миграции import-style в существующих тестах без отдельной задачи.

## Stories

- Для новых компонентов и существенно переписанных story-файлов используй modern pattern из `docs/ai/stories-guide.md`: `examples/`, `?raw`, копируемые примеры.
- При небольших изменениях legacy stories не рефакторь файл только ради миграции структуры; сохраняй локальный паттерн и story ids.
- При добавлении нового prop: обновить Playground. Если prop визуальный — добавить named story или покрыть его существующей visual story.
- Примеры для копирования должны импортировать публичный API из `@sberbusiness/triplex-next` или публичных subentry, а не `src/...`.

## Breaking Changes

- Переименование props, enum-значений, публичных CSS-переменных — это breaking change (мажорная версия).
- Смена ref-target, корневого DOM-элемента, story id, используемого в e2e/visual flows, или публично документированного accessibility-контракта тоже рассматривается как потенциально breaking.
- Breaking changes описываются в PR description.
- Deprecated API — с JSDoc `@deprecated` и указанием альтернативы.

## Принципы

- **DRY**: не дублируй код (исключение: examples в stories).
- **KISS**: простейшее решение, которое работает.
- **YAGNI**: не добавляй то, чего нет в текущем требовании.
- Не добавляй комментарии/docstring к коду, который не изменял.
- Не рефакторь то, что не просили трогать.
