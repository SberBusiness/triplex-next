# AI refactoring

Правила, по которым AI-агент рефакторит существующие компоненты при работе по
колонке **AI refactoring** в `docs/ai/ROADMAP.md`.

Цель — сделать код чище, единообразнее и удобнее для последующих AI-правок,
**не ломая публичный API**. Если есть сомнение, можно ли менять что-то без
breaking change — оставь как есть.

---

## Воркфлоу

1. **Аудит.** Прочитай ВСЕ файлы папки компонента: `*.tsx`, `*.ts`, `styles/*.module.less`, `__tests__/*`, и связанные stories. Отметь, какие субкомпоненты не покрыты unit-тестами. Не делай выводы по одному файлу.
2. **План.** Выпиши конкретные правки в формате `файл:строка → что → почему`. Раздели на категории (codestyle / structural / AI-friendliness / unit-тесты — см. ниже). Явно перечисли, что НЕ трогаешь и почему.
3. **Согласование.** Покажи план пользователю. Не правь код до согласования.
4. **Применение.** Делай правки последовательно, мелкими шагами, в порядке плана. Тесты добавляй последним этапом — после всех code-правок, чтобы зафиксировать итоговое поведение.
5. **Проверка.** Прогоняй три инструмента и чини всё, что относится к файлам компонента:
   - `npx eslint src/components/{Name}` — 0 ошибок и 0 warnings в затронутых файлах. ESLint в этом проекте включает строгие правила `react-hooks` (`set-state-in-effect`, `refs`, `immutability`, `exhaustive-deps`), которые ловят антипаттерны, не видимые `tsc`. Если что-то ругается на твой рефакторинг — фикси на уровне архитектуры (см. примеры ниже), а не глуши `eslint-disable`. `eslint-disable` допустим только если есть неустранимое объяснение в комментарии рядом.
   - `npx tsc --noEmit` — отфильтруй ошибки по путям компонента, посторонние pre-existing ошибки игнорируй.
   - `npx vitest run src/components/{Name}` — все тесты зелёные. Если добавлял тесты — проверь, что новые файлы попали в прогон.

   Типичные ошибки `react-hooks`, которые встречаются после рефакторинга:
   - `react-hooks/set-state-in-effect` — `setState` синхронно внутри `useEffect`. Чини через `useState` lazy initializer, derived state или вынос в event handler.
   - `react-hooks/refs` — чтение `ref.current` во время рендера. Чини через `useState` (если значение нужно в JSX) или через `useImperativeHandle`.
   - `react-hooks/immutability` — обращение к функции-замыканию до её объявления (в callback'е useEffect, объявленного выше функции). Чини перестановкой блоков или инлайнингом.
6. **ROADMAP.** Поставь ✅ в колонке `AI refactoring` для компонента.

## Три категории изменений

### 1. Codestyle — привести к канонам `docs/ai/codestyle.md`

Типичные находки:

- **Рассинхрон `displayName` и имени компонента** (например, после переименования): `MyComponent.displayName = "OldName"` → `"MyComponent"`. То же для JSDoc-комментариев типа `/** Свойства компонента OldName. */`.
- **Inline-комментарии `// ...` на полях интерфейса** → JSDoc `/** ... */`. JSDoc парсится IDE и AI-агентами, обычные комментарии — нет.
- **`typeof x !== "undefined"`** — заменять только осознанно, не «всё подряд». Этот чек проходит для `0`, `""`, `false`, `null`, поэтому слепая замена на `Boolean(x)` меняет поведение на falsy-значениях. Безопасные варианты:
  - **`x !== undefined`** — точный эквивалент, если нужно отличать `undefined` от любого определённого значения (включая `null`, `0`, `""`, `false`).
  - **`x != null`** — если по контексту корректно объединить `undefined` и `null` (типичный случай для опциональных props).
  - **`Boolean(x)`** — только когда осознанно отфильтровать все falsy значения (`0`, `""`, `false`, `null`, `undefined`) — это и есть желаемое поведение. Применимо для строгих nullable/optional случаев, где falsy-значения семантически отсутствуют.
  - **Явные проверки** (`x === 0`, `x === false`, `x === ""`) — если такие значения должны сохраняться и обрабатываться отдельно от `undefined`/`null`.

  Применяется в условиях `clsx`/тернарниках. Перед заменой убедись, что тип `x` исключает значимые falsy-значения, иначе оставь `typeof`-чек либо используй явный guard.
- **Мёртвые CSS-селекторы**: класс, который НИГДЕ не выставляется (ни в TSX, ни через `className` снаружи — для CSS-модулей это безопасно проверить grep'ом). Удалить.
- **Noise-комментарии**: `/** Функция для хранения ссылки. */` над функцией `setRef` — не несут смысла. Удалить.
- **`!` non-null assertion** без комментария-обоснования — переписать через type guard или `Boolean(x)`.

### 2. Structural — упростить структуру

- **Дедупликация между sibling-компонентами в той же папке.** Пример: два сенсора DnD с идентичным DOM-обходом → вытащить общий хелпер `isDraggableTarget(target)`. Два разных компонента с одинаковой merge-ref логикой → общий `setForwardedRef(ref, instance)`.
- **Локализация хелперов.** Внутренние утилиты компонента кладутся в `src/components/{Name}/utils.ts` (или подобное) и НЕ экспортируются через barrel `index.ts`. Не добавляй в `src/utils/index.ts` — это публичный API библиотеки.
- **Порог YAGNI.** Не выноси хелпер ради 1–2 мест с 2–3 строчками. Извлекай, когда есть ≥2 места и ≥5 строк дублирования, либо когда логика концептуально единая (например, обход DOM по data-атрибуту).
- **Не добавляй новые абстракции.** Хуки, контексты, паттерны — только если они уже устраняют конкретное дублирование. Никаких «на будущее».
- **Чистые initializer/factory-функции — в module scope.** Если `useState(() => {...})`, `useRef(() => {...})`, `useMemo(() => {...})` или `useCallback` содержат тело ≥5 строк, которое **не замыкает** ни props, ни state, ни другие локальные переменные компонента (ссылается только на module-level константы и глобалы) — вынеси функцию на module scope с говорящим именем. Внутри хука остаётся одна понятная строчка: `useState(getOrCreateMountNode)`. Это разгружает тело компонента и делает initializer независимо тестируемым. Пример: `getOrCreateMountNode` в `ModalWindow.tsx`, `getOrCreateModalWindowViewManagerNode` в `ModalWindowViewManager.tsx`. Если функция замыкает на `props`/`state` — оставляй inline (вынос потребует протаскивать аргументы, и это не упростит, а усложнит).

### 3. AI-friendliness — улучшить читаемость для агентов

- **JSDoc на каждом публичном prop.** Интерфейсы, экспортируемые через barrel — часть публичного API. Все их поля должны быть задокументированы. Минимум — одна строка по делу.
- **JSDoc на компоненте**: одно-два предложения, что делает.
- **Говорящие имена приватных хелперов.** `setRef` локально OK; в общем хелпере — `setForwardedRef`.
- **Удалить мёртвый код**: неиспользуемые типы, функции, классы, импорты, селекторы.

## Покрытие unit-тестами

При AI-рефакторинге часто оказывается, что у компонента или его субкомпонентов
нет unit-тестов. Это окно возможностей: пока ты глубоко в коде и понимаешь
поведение — зафиксируй его тестами. Они дают страховку для будущих
рефакторингов и AI-правок.

Общие правила тестирования — `docs/ai/tests.md`. Здесь — что специфично для AI-рефакторинга.

### Когда добавлять

- Если рефакторинг затронул нетривиальную логику (вытащил хелпер, дедуплицировал поведение, поменял хук) — обязательно покрыть её тестом.
- Если в папке компонента есть субкомпоненты без тестов с публичным API — добавь тесты в рамках задачи.
- Если правки чисто косметические (JSDoc, переименование noise-комментария, удаление мёртвого селектора) — не обязательно, но прогон существующих тестов обязателен.

### Что покрываем — приоритет

**Высокий — нетривиальная логика, ловит регрессии:**
- Pure-функции и утилиты компонента (`utils.ts`) — каждая ветка, включая edge cases (null, не-HTMLElement и т.п.).
- Контекст и синхронизация состояния между субкомпонентами (например, `selected` через React context от `Selectable` к `Content`).
- Клавиатурные обработчики (`Enter`/`Space`/`Arrow*`, `preventDefault`).
- ARIA-атрибуты, кодирующие состояние (`aria-expanded`, `aria-haspopup`, `aria-controls`).
- Discriminated union props — обе ветки рендерятся корректно.

**Средний — публичный контракт компонента:**
- Все значения enum-prop → корректный класс / атрибут.
- Комбинаторные классы (`withIcon`/`withText` от наличия пропов).
- Колбэки (`onChange`/`onClick`/`onSelect`) вызываются с правильными аргументами, а не только сам факт вызова.
- `disabled` блокирует обработчики.
- `forwardRef` пробрасывается на ожидаемый DOM-элемент.
- `className` мерджится в корневой элемент.

**Низкий — тривиальные обёртки:**
Компоненты, делающие только `clsx + className + ref + ...rest`. Если уже
пишешь тесты для других компонентов в этой папке — добавь smoke-тест (рендер +
ref + className) для единообразия. В одиночку — пропускай: тесты на
тривиальное не ловят регрессий.

### Что НЕ тестируем unit-тестами

- **Сторонние интеграции** (DnD-сенсоры `@dnd-kit`, портальные dropdown-механики, реальный drag-and-drop). Если логика, которую тебе нужно проверить, вытащена в чистую функцию (например, `isDraggableTarget`) — тести функцию. Сам drag — это E2E.
- **Визуальное отображение** — visual regression (Storybook test-runner).
- **Pre-existing баги.** Если по ходу заметил подозрительное поведение (например, `EVENT_KEYS.SPACE = ["Spacebar", "Space"]` может не матчить современный `event.key === " "`) — отметь это в плане как находку, но не «случайно фиксируй» багованное поведение зелёным тестом.

### Куда класть тесты

`src/components/{Name}/__tests__/{Subcomponent}.test.tsx` — **отдельный файл на каждый
субкомпонент с нетривиальной логикой**. Не сваливай 50 тестов в один `{Name}.test.tsx`.

Расширение `.test.tsx` обязательно даже для тестов чистых функций без JSX
(`utils.test.tsx`) — `vitest.config.ts` включает только `src/**/*.test.tsx`.

CSS-модули в тестах не хешируются (`classNameStrategy: "non-scoped"`), поэтому
`toHaveClass("selected")` работает напрямую с именем класса из `.module.less`.

### Минимум, что должно быть в тестовом файле

- Рендер с обязательными props.
- Все варианты enum-props → корректный класс или атрибут.
- Каждый колбэк → проверка аргументов вызова (`toHaveBeenCalledWith(...)`, не просто `toHaveBeenCalled`).
- `forwardRef` → `expect(ref.current).toBeInstanceOf(HTMLXxxElement)`.
- `className` → проверка, что кастомный класс есть на корневом элементе.
- Нетривиальные ветки логики (keyboard-обработчики, контекст, conditional rendering).

## Изменения публичного API — спроси подтверждение пользователя

Всё перечисленное ниже — breaking change для потребителей библиотеки. **Не правь
самостоятельно.** Если в ходе аудита нашёл что-то, что хочется починить (например,
рассинхрон имени интерфейса с именем компонента после прошлого переименования) —
добавь это в план как **отдельный пункт «требует подтверждения»**, явно укажи
последствия (breaking change, нужна запись в release notes, при необходимости —
мажорная версия) и дождись согласия пользователя.

После подтверждения — правь напрямую (без re-export-алиасов и backwards-compat
обёрток) и обнови `stories/release-notes/v1/<следующая-версия>.mdx`.

- **Имена экспортируемых интерфейсов / типов / enum'ов / классов**, попадающих в barrel `index.ts`. Пример: `IListItemEmptyProps` → `IListEmptyStateProps` (рассинхрон с именем компонента после прошлого переименования).
- **Имена и сигнатуры экспортируемых компонентов и их props.** Любое переименование, изменение типа, изменение `default`.
- **Значения enum'ов** (строковые/числовые литералы).
- **Barrel exports.** Удаление любого экспорта из `index.ts`.
- **`forwardRef`** на компонентах — обязателен, не убирать.
- **Корневой DOM-элемент и ref-target** публичного компонента.
- **CSS-переменные дизайн-токенов** (`--triplex-next-...`) — переименование ломает темизацию у потребителей.
- **Имена LESS-классов**, которые могут использоваться снаружи. Для CSS-модулей класс хешируется, и потребитель не может опираться на имя из JS — но имя всё равно может фигурировать в e2e/visual-тестах или в `className.includes(...)`. Перед переименованием проверь grep'ом.
- **`displayName`** — обычно часть публичного API (React DevTools, snapshot-тесты). **Исключение:** рассинхрон с именем компонента после переименования — это явный fix, можно править без отдельного подтверждения, но упомяни в плане.
- **Story IDs**, на которые опираются e2e/visual-тесты — не переименовывать без синхронного обновления тестов.

## Особые ограничения проекта

### React 17 совместимость через ветку release-0

Ветка `release-0` поддерживает legacy v0.x на React 17 и периодически синхронизируется
мейнтейнерами с `main`. Поэтому в коде компонентов **не используй React 18-only API**:

- `useId` (используй `useRef(uniqueId())` или `useState(() => uniqueId())` из `lodash-es` / `scripts/uniqueId`)
- `useSyncExternalStore`
- `useInsertionEffect`, `useTransition`, `useDeferredValue`
- Поведение, зависящее от automatic batching React 18

Это касается рефакторинга — не мигрируй существующий React 17-safe код на React 18 API.
Исключения возможны (есть компоненты, использующие `useId`), но это решение мейнтейнера, а не дефолт AI-рефакторинга.

`docs/ai/CONTEXT.md` упоминает `React.useId()` как «предпочтительно» — это
относится к коду, который точно не уйдёт в `release-0`. При AI-рефакторинге исходи из
React 17-совместимости по умолчанию.

## Out of scope для AI refactoring

Не делай в рамках этой задачи, даже если видишь возможность:

- **Миграция legacy stories / тестов** на modern pattern (см. `docs/ai/stories-guide.md`). Новые тесты пиши в modern-стиле, старые не переписывай.
- **Изменение публичного API без подтверждения** — см. раздел выше. С подтверждением — допустимо.
- **Фикс type errors в чужих файлах** (не в папке компонента). Если `tsc` показывает ошибки — отфильтруй по путям компонента.
- **Добавление новых props / вариантов.** AI-рефакторинг — про код, а не про функциональность.
- **Добавление утилит в `src/utils/index.ts`** или другие публичные barrel-файлы.

## Чек-лист перед завершением

**Публичный API:**
- [ ] Публичный barrel `index.ts` компонента не изменился, либо изменения — переименования / удаления — согласованы с пользователем.
- [ ] Все breaking-изменения имён попали в `stories/release-notes/v1/<следующая-версия>.mdx` с пометкой **Breaking**.
- [ ] `forwardRef` на месте у всех публичных компонентов.

**Код:**
- [ ] `npx eslint src/components/{Name}` — 0 ошибок и 0 warnings в файлах компонента. Все `react-hooks/*` нарушения исправлены (см. шаг «Проверка»), `eslint-disable` использован только с комментарием-обоснованием.
- [ ] `npx tsc --noEmit` — 0 ошибок в файлах компонента (отфильтрованы посторонние pre-existing ошибки).
- [ ] Не использован React 18-only API (если только не было до рефакторинга).

**Тесты:**
- [ ] `npx vitest run src/components/{Name}` — все тесты зелёные.
- [ ] Все субкомпоненты с нетривиальной логикой имеют unit-тесты (см. раздел «Покрытие unit-тестами»).
- [ ] Если рефакторинг вытащил новый хелпер / хук / утилиту — у него есть собственный тестовый файл.
- [ ] Колбэки тестируются на содержимое аргументов (`toHaveBeenCalledWith(...)`), не только на факт вызова.

**Финал:**
- [ ] `docs/ai/ROADMAP.md` — поставлен ✅ в колонке `AI refactoring`.
- [ ] В отчёте пользователю явно перечислено: что менялось, что НЕ менялось и почему, какие тесты добавлены.
